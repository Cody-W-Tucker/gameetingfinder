Executive summary

Core finding

Most of the resilience work is started but we're missing a target for a full recovery system built.

Main risks

Local disk writes, ad hoc backups, missing health visibility, and origin pressure from admin and analytics behavior.

How to use this

Each section points to a specific place in the repo so someone can pick up the work without digging through the codebase again.

1. Current resilience shape

The app fairly portable. JWT cookies mean the web tier is not tied to a single process. Postgres holds the business state. Static assets are already cacheable.

Observed strengths

  • `web/auth.py` and `web/main.py` use JWT cookies, not server-memory sessions
  • `web/main.py` reads business state from Postgres, not local memory
  • `web/main.py` sets long cache headers for `/static/*`
  • `web/main.py` already treats anon meeting data as cacheable for short windows

Why these are problematic

  • Local avatar writes: avatars stored in `web/static/avatars/` disappear on container restart, can't be shared across multiple app instances, and create state that lives outside the database—breaking the portability.
  • Local invoice directory: Same as above, plus these would be missing if, for example, you spun this up in a different server.
  • Missing `/healthz`: without a health endpoint, load balancers can't route away from failing instances, monitoring can't auto-alert. So you wouldn't know if the site is down and you have no option for automatically failing over to another server.
  • Local-script backups: `backup_db.py` writes to the same disk it's protecting, requires manual execution, and provides zero protection against the most common failure mode—loss of the host itself
Implementation note: the app is fairly portable. The real portability problems are file storage, recovery procedure, and operational coupling.

2. Recovery path

The repo shows parts of a recovery path, but not one path that the team can trust end to end. There is enough here to rebuild the app. What is missing is the sequence that turns a failure into a routine restore instead of an improvisation.

Observed recovery helpers

  • `Dockerfile` and `Procfile` show a clear runnable app shape
  • `migrate.py` gives a straightforward schema migration path
  • `sync_local.py` shows the team already thinks in terms of restoring DB state from dumps
  • Most public behavior can be recreated from app code plus database state

Observed recovery gaps

  • No repo-level runbook: DB restore, env, and host rebuild steps exist separately but aren't tied together, so recovery requires tribal knowledge and improvisation instead of being a repeatable procedure
  • Runtime files not externalized: avatars and invoices still live on local disk, so they're lost during recovery and can't be restored from DB backup alone
  • Recovery success checks implied: there's no codified checklist of "what working looks like," so the team can't confidently declare recovery complete
Implementation note: you're missing a restore path somone else could run without tribal knowledge.

3. Backup and data durability

The repo already makes it clear that the current backup posture is temporary. The question is not whether someone can dump the database. The question is whether backups, retention, off-platform storage, and restore confidence are treated as part of normal operations.

Observed in the repo

  • `backup_db.py` writes compressed dumps to a local `backups/` folder
  • `backup_db.py` assumes Windows paths and scheduled local execution
  • `CONTRIBUTING.md` explicitly calls out off-Railway backup storage as needed work
  • `sync_local.py` shows restore is possible, but not yet treated as production recovery discipline

What needs to be explicit

  • Which data sets are mandatory to preserve
  • Where DB backups live outside the host and outside the provider
  • Which runtime files also need durable storage, not just DB backup
  • What a successful restore check looks like after recovery
Implementation note: the backup gap is simple: there is a script, but there is not yet a recovery system.

4. Uptime pressure inside the app

The clearest uptime clue is already in the comments. This app has seen origin stress from internal behavior, not just provider outages. Resilience work should start by reducing self-inflicted load.

Observed in the repo

  • `web/main.py` adds TTL caches for hot admin endpoints
  • `web/main.py` comments explicitly mention ~8.7k HTTP 504s from admin polling load
  • `web/main.py` pre-pings DB connections to avoid stale-connection failures
  • `web/main.py` uses short-lived in-process caches for meetings and visitor counts

What should improve first

  • Separate public-site stability from `/dailypaper/*` polling and analytics pressure
  • Identify endpoints currently protected only by in-process caching
  • Add explicit health visibility instead of inferring health from page loads
  • Decide which internal surfaces are allowed to degrade before the public site does
Implementation note: some of the uptime problem is already diagnosed in comments. The next step is to treat those comments as design constraints instead of leaving them as history.

5. Business continuity

This is where technical recovery becomes something people can actually use. Portable code and existing scripts are not enough. Someone still needs to know what to restore first, what counts as "good enough," and which parts of the system can wait.

What needs to be clear

  • Who owns recovery when the app, DB, or host fails
  • What gets restored first: public site, auth, admin, analytics
  • Which services are required for "good enough" restoration
  • How the team verifies that the site is safe to trust again

What needs a decision

  • Who decides the site is "restored enough" to reopen fully
  • What gets deferred during an incident instead of fixed live
  • What communication goes out when service is degraded
  • What recovery steps are important enough to document now
Implementation note: this is the point where a technical recovery path turns into an operating procedure.
